# blitz perf patterns

## Correction to current spec

- `ts_parser_parse` arity is 3, not 4: `ts_parser_parse(TSParser *self, const TSTree *old_tree, TSInput input)`. `ts_parser_parse_string` is the 4-arg helper for contiguous buffers. For repeated edits to same file, edit current tree with `ts_tree_edit` before each reparse, keep byte + point deltas exact, and re-fetch cached `TSNode`s or call `ts_node_edit` after tree mutation. Source: https://raw.githubusercontent.com/tree-sitter/tree-sitter/master/lib/include/tree_sitter/api.h
- `~2-5 ms` cold-start / parse-time claim is too hot for medium files. Public rewrite benches are much higher: ast-grep reports `0.975s` total for a TypeScript/src scan after optimizations and `43ms` on a 500-file rewrite batch; Comby FAQ shows `187ms` on a 2,591-line Go file; srgn docs show `~1s` for 450k Python lines and `~3s` for ~3M Go lines. Treat blitz latency numbers as targets, not facts. Sources: https://ast-grep.github.io/blog/optimize-ast-grep.html ; https://comby.dev/docs/faq ; https://docs.rs/crate/srgn/latest
- Coverage / savings bands in `blitz-design.md` and `blitz-gap-closure.md` are not evidence-backed yet. No public corpus shows `45-55%` handled-case savings, `~93-97%` real-workload coverage, or `~3-7%` residual miss for this exact stack. Keep those as benchmark hypotheses until measured on same edit set as fastedit. Sources: https://docs/architecture/blitz-design.md ; https://docs/architecture/blitz-gap-closure.md
- LOC estimates are rough, not contracts. `Continue`, `Cline`, and `Aider` all need extra scaffolding around matching, prompt shaping, validation, and fallback, so `250-500` / `700-1200` should stay as ranges only. Sources: https://raw.githubusercontent.com/continuedev/continue/main/core/edit/lazy/replace.ts ; https://raw.githubusercontent.com/cline/cline/main/src/core/prompts/system-prompt/tools/apply_patch.ts ; https://raw.githubusercontent.com/Aider-AI/aider/main/aider/coders/search_replace.py
- `Continue` splice path is exact-context first, not fuzzy-first: `replace.ts` matches 1 line above + up to 3 below; `deterministic.ts` uses `fastest-levenshtein` for AST-node recovery elsewhere. So do not label its splice engine as pure Levenshtein matching. Source: https://raw.githubusercontent.com/continuedev/continue/main/core/edit/lazy/replace.ts ; https://raw.githubusercontent.com/continuedev/continue/main/core/edit/lazy/deterministic.ts
- `diff-match-patch` is Myers diff + Bitap match, not magic merge. Upstream README says it implements Myers diff, pre/post cleanup, and flexible matching/patching; classic Myers SES is O(ND), so keep DMP as fallback, not core first-pass splice. If you need a text diff alternative, Git's `--diff-algorithm=myers|histogram` docs give a clearer baseline for deterministic splice heuristics. Sources: https://raw.githubusercontent.com/google/diff-match-patch/master/README.md ; https://neil.fraser.name/writing/diff/myers.pdf ; https://git-scm.com/docs/git-diff
- Cursor public docs expose selection-based inline edit / Agent, but no published `# ... existing code ...` marker grammar or splice internals. Do not assume marker-aware behavior there. Sources: https://cursor.com/docs/inline-edit/overview ; https://cursor.com/help/ai-features/agent

## Top 5 perf patterns to adopt

1. **Incremental parse reuse.** Parse once, keep current tree, then apply `ts_tree_edit` + reparse with previous tree. That lets unchanged subtrees survive and keeps work local. If you cache node handles, edit those too or re-fetch from edited tree. Source: https://raw.githubusercontent.com/tree-sitter/tree-sitter/master/lib/include/tree_sitter/api.h
2. **Cache queries, reuse cursors, shrink range.** Build `TSQuery` once per language+pattern, keep `TSQueryCursor` hot, and narrow execution with `ts_query_cursor_set_byte_range`, `ts_query_cursor_set_containing_byte_range`, `ts_query_cursor_set_point_range`, and `ts_query_cursor_set_max_start_depth`. Query objects are immutable; cursors can be reused. Source: https://raw.githubusercontent.com/tree-sitter/tree-sitter/master/lib/include/tree_sitter/api.h
3. **Deterministic anchor ladder.** Aider-style pipeline is good baseline: exact search/replace, relative-indent normalization, blank-line stripping, DMP patching, then git-style minimal merge. Relative-indent transform lowers token cost because unchanged body survives as compact structure instead of repeated text. Source: https://raw.githubusercontent.com/Aider-AI/aider/main/aider/coders/search_replace.py
4. **Byte-level fuzzy recovery.** For anchor windows, prefer byte-level edit distance on UTF-8 when input is ASCII-ish; switch to codepoint-aware scoring only when needed. RapidFuzz docs explicitly say bytes are faster for ASCII-only inputs, and uniform Levenshtein uses bit-parallel `O([N/64]M)` behavior. Source: https://rapidfuzz.github.io/RapidFuzz/Usage/distance/Levenshtein.html ; https://docs.rs/rapidfuzz/latest/rapidfuzz/
5. **Structured rewrite IR, not free-text rewrite.** Compile patch ops into tree-sitter query captures / Comby templates / ast-grep `pattern+rewrite` so edit target is a node span, not a whole file. That cuts prompt tokens and lets engine stay deterministic. Sources: https://tree-sitter.github.io/tree-sitter/using-parsers/queries/1-syntax.html ; https://comby.dev/docs/api ; https://ast-grep.github.io/guide/rewrite-code.html

## Token-efficiency prompt patterns

- **Use lazy edit markers.** Tell model to output only changed lines plus `// ... existing code ...`-style keep markers, never full-file rewrite. Morph, Relace, and similar fast-apply docs all rely on this exact contract. Sources: https://docs.morphllm.com/api-reference/endpoint/apply ; https://docs.relace.ai/docs/instant-apply/quickstart
- **Put intent in 1 short sentence, then code.** Best prompt shape: short first-person instruction, then original code, then abbreviated update snippet. That keeps instruction entropy low and makes merge step deterministic. Sources: https://docs.morphllm.com/api-reference/endpoint/apply ; https://docs.relace.ai/docs/instant-apply/workflow
- **Keep deletion syntax explicit.** If omitting a span means delete it, say so. Relace docs show `// ... keep existing code ...` and explicit delete context; Cline warns that omitted context can look like truncation. Sources: https://docs.relace.ai/docs/instant-apply/quickstart ; https://raw.githubusercontent.com/cline/cline/main/src/integrations/editor/detect-omission.ts
- **Preserve exact indentation in snippet.** Prompt should say preserve indent + final structure exactly, so merge engine can splice without reflow. This matters for relative-indent and patch-based merge paths. Sources: https://docs.relace.ai/docs/instant-apply/workflow ; https://raw.githubusercontent.com/Aider-AI/aider/main/aider/coders/search_replace.py
- **Use snippet grammars as compression DSLs.** Good minimum-output formats: tree-sitter query captures, Comby `match/rewrite`, ast-grep `pattern` + `rewrite` / YAML `fix`, or Relace `initial_code + edit_snippet`. These let model emit patch IR, not regurgitated source. Sources: https://tree-sitter.github.io/tree-sitter/using-parsers/queries/1-syntax.html ; https://comby.dev/docs/basic-usage ; https://ast-grep.github.io/guide/rewrite-code.html ; https://docs.relace.ai/api-reference/instant-apply/apply
- **Cursor note.** Cursor public docs show selection-based inline edit / Agent context, so use prompt patterns that work even without marker parsing. Sources: https://cursor.com/docs/inline-edit/overview ; https://cursor.com/help/ai-features/agent
